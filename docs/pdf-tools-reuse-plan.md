# Reusing from Open-Document-Alliance/PDF-Tools

## 1. Summary

Open-Document-Alliance/PDF-Tools is a privacy-first, desktop-local PDF toolkit built as a Node.js MCP server (43 tools) plus a Vite-compiled single-file embedded viewer, designed to run inside Claude Desktop. It is MIT licensed. The headline of what is worth reusing is conceptual, not code: most of its implementation is either Node-only (`@napi-rs/canvas`, MCP SDK, `fs`-based persistence) or already present in our codebase. The highest-value candidates are self-contained UX features that can be built from libraries we already ship (pdf-lib, pdfjs-dist, xlsx, zustand) with no new dependencies. The MCP server layer, `@napi-rs/canvas`, `vite-plugin-singlefile`, and all stateless-server architectural patterns are incompatible with our browser-only React 19 + Vite architecture and should be skipped entirely.

> **License caveat:** MIT license on the source. No adoption blocker, but the repo's actual implementation files (`server/index.js`, `ui/`) were not directly read — the recon is based on README + package.json only. Any server-side utility logic confirmed below as "adapt" should be manually inspected before porting to verify no undisclosed dependency or license constraint is embedded.

## 3. Investigate Further

| Candidate                                 | Open Question                                                                                                                                                                                                                                                                                                                                                                                                      |
| ----------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Password-protected PDF unlock UX**      | react-pdf's `Document` already calls `window.prompt()` via its default `onPassword` — so PDFs are not silently failing. The real question: is replacing `prompt()` with an in-app modal + fixing the unguarded `pdfjs.getDocument()` in the OCR path worth prioritising now, or is it low-enough frequency to defer? Manually confirm what happens today when a password-protected PDF is dragged in, then decide. |
| **Bulk fill from CSV**                    | Depends entirely on AcroForm fill (item 9) being shipped first. Validate: (a) do target users actually have PDFs with AcroForm fields (not XFA)? (b) is there measurable demand for the HR/legal batch use case specifically? Do not build until item 9 ships.                                                                                                                                                     |
| **Page-level blank/orientation analysis** | Orientation is free from `usePageHeights` data but has no UI surface. Blank-page detection adds a per-page render pass (cost) in exchange for smart-split suggestions (speculative value). Investigate whether a page-metadata tooltip or smart-split feature is concretely planned before paying the render cost.                                                                                                 |
| **Smoke tests / Playwright**              | The gap is real (zero component or integration tests). The open question is bootstrapping cost: the Vite dev server needs COOP/COEP headers for `SharedArrayBuffer`, WebGPU must be mocked or a CPU fallback flag used for Florence-2, and PaddleOCR WASM needs a CDN intercept. Spike one Playwright test against a fixture PDF to validate the CI setup before committing to a full suite.                       |

## 4. Skip / Already Covered

| Candidate                                            | Reason                                                                                                                                                                                                                        |
| ---------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `parsePageRanges` / `mergePdfs` utility              | Identical implementations already in `src/lib/mergeSplitPdf.ts` with passing unit tests.                                                                                                                                      |
| `@modelcontextprotocol/sdk` + MCP server scaffolding | Node.js server-side SDK, architecturally incompatible with a browser-only Vite app. We have no backend.                                                                                                                       |
| `@napi-rs/canvas`                                    | Native Node.js C++ addon; cannot run in a browser. We already have `HTMLCanvasElement` and `OffscreenCanvas` natively.                                                                                                        |
| `vite-plugin-singlefile`                             | Breaks our Worker architecture (browsers forbid constructing Workers from `data:` URIs). The 74MB dist dominated by ONNX blobs makes single-file inlining impractical regardless. GitHub Pages deployment already works fine. |
| MCP tool-driven stateless architecture               | Runtime constraint of Claude Desktop MCP hosts, not a UI pattern. Adopting it means dismantling zustand and undo history — a regression, not an improvement.                                                                  |
| pdfjs-dist patch upgrade (5.4.296 → 5.4.624)         | react-pdf@10.4.1 hard-pins pdfjs-dist at exactly 5.4.296. Upgrading independently breaks the Worker/API pairing. Wait for a react-pdf release that bumps the pin.                                                             |
| `vitest` (standalone)                                | Already present via `@voidzero-dev/vite-plus-test`. Adding standalone vitest would risk a version conflict.                                                                                                                   |
| Form field sidebar (standalone)                      | Has zero value without the AcroForm fill feature built first. Build item 9; add the sidebar as a refinement pass.                                                                                                             |
| Reusable personal/business profile store             | Depends on AcroForm fill infrastructure that does not exist. Also introduces XSS-exfiltration risk for SSN/address data stored in localStorage that the candidate hand-waves. Skip until form fill ships.                     |
| Single-entry action registry pattern                 | Already ~80% present: `useEditorActions.ts` + `CommandPalette.tsx` `PaletteAction` type covers this. The only gap is a ~30-line keyboard shortcut refactor in `App.tsx`, not an architectural adoption.                       |

## 5. Suggested Order of Work

**Phase 1 — Quick wins (no prerequisites, low effort):**

1. Saved signatures library (item 1) — closes an immediately visible UX gap for repeat signers.
2. Interval-based PDF split (item 2) — purely additive, touches only `mergeSplitPdf.ts` and `SplitDialog.tsx`.
3. Structured text extraction to CSV (item 3) — ~30 lines, uses only existing deps.
4. PDF metadata panel (item 4) — one utility + one modal, zero risk.
5. Zustand persist for settings (P1) — one `persist` wrapper, no new dep.

**Phase 2 — Medium-effort UX improvements (parallelisable):** 6. URL-to-PDF open — small but requires deliberate CORS error UX. 7. Page-level image detection + auto-OCR prompt — improves discoverability for the most-used feature. 8. Visual page reorder polish — low-risk CSS/state improvement.

**Phase 3 — Investigation spikes (validate before building):** 9. Password unlock UX — confirm current behavior in the app before deciding scope. 10. Playwright smoke test bootstrapping — spike one test to validate CI setup cost. 11. Signature zone auto-detection — prototype the regex pass on real-world legal PDFs to validate hit rate before building the overlay layer.

### Phase 3 — Outcomes (implemented test-first)

All three spikes were implemented test-first; `pnpm test` (118 unit tests) and the Playwright smoke test pass.

- **Signature zone auto-detection (item 11).** `src/lib/signatureZones.ts` — pure `classifyLabel()` (word-anchored regex for sign/initials/date, so "designation"/"candidate" don't match) + `findSignatureZones()` placing a fitted target to the right of each label, with `resolveImagePlacement()` for aspect-preserving drop-into-zone. Wired via a new `"signZones"` editor mode + `SignatureZoneLayer.tsx` overlay (mirrors `ExistingTextLayer`); clicking a zone records a `signaturePlacement` the `SignatureModal` consumes so the signature lands in the zone. Command-palette entry "Find signature spots". Tests: `signatureZones.test.ts` (11).
- **Password unlock UX (item 9).** Confirmed: react-pdf's `<Document>` prompted via native `window.prompt()`, while the direct `pdfjs.getDocument()` paths (CSV export, scanned-detection, page-height) had NO password handling and threw silently. Fix: `src/lib/pdfPassword.ts` pure helpers (`isPasswordException`, `passwordPromptState`, `makeOnPassword`) + a shared `loadPdfDocument()` in `pdfOptions.ts` that feeds the stored password to all background paths (no duplicate prompt — the viewer's `<Document onPassword>` is the single prompt surface). New `PasswordModal.tsx` replaces `prompt()`; submitting re-keys `<Document>` to reload decrypted. Tests: `pdfPassword.test.ts` (8).
- **Playwright smoke-test bootstrap (item 10).** Validated CI cost is low. Added `@playwright/test` + Chromium, `playwright.config.ts`, `e2e/global-setup.ts` (generates a fixture PDF with pdf-lib — no binary committed), and `e2e/smoke.spec.ts` (boots the app, asserts `crossOriginIsolated`, opens the fixture via the real file input, asserts a page canvas renders). COOP/COEP headers added to the Vite dev+preview server (`server.headers`/`preview.headers`) so `SharedArrayBuffer` works locally and under test. e2e specs excluded from the vitest runner (`test.exclude`). Run with `pnpm test:e2e`. **Note:** `vite`/`vite-plus` were pinned from `latest` → `0.1.24` because adding Playwright re-resolved `latest` to a 0.2.x that broke the test runner; pinning also makes builds reproducible. WebGPU/PaddleOCR-CDN mocking was NOT needed for the smoke test (it doesn't run OCR) — defer that to whenever an OCR e2e test is added.

**Phase 4 — High-effort foundational feature:** 12. AcroForm fill (item 9, adapt recommendation C) — the largest single investment. Do not start until Phase 1 and 2 are complete. Scope to single-document interactive mode first; bulk CSV fill and the form field sidebar are follow-ons that depend on this foundation.
